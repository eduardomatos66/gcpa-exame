  • C. Granting iam.serviceAccountUser to the administrator at the service account level

Option A is correct. A user can manage all the service accounts in the project if service account user role iam.serviceAccountUser is granted to him at the project level. Also, whenever a new service account will be created, the administrator will be granted iam.serviceAccountUser automatically for that new service account.

Option B is incorrect. iam.serviceProjectAccountUser is not a service account user role in Google Cloud.  

Option C is incorrect. Granting iam.serviceAccountUser to the administrator at the service account level means the service account user role is granted for a specific account. It will enable him to manage that specific service account in the project, not all the service accounts. It will require setting the role for all service accounts separately. Also, If a new service account will be created, the administrator will have to grant iam.serviceAccountUser to the other administrator.

Question 3Correct
Domain: Ensuring successful operation of a cloud solution

You’re working for a company as a Cloud engineer. You have a managed instance group for which the autoscaling is enabled if CPU utilization is 70%. Right now there are 4 instances available in this instance group. When you connect to one of the instances, you see that the CPU utilization is 75% and there is no new instance is started in the group. What could be the possible reason for not starting another instance here?

  • A. It will take 60 seconds to launch a new instance
  • B. Autoscaling is not active
  • C. The average CPU usage of the instance group is less than 70%right
  • D. None of the above
Explanation:

The correct answer is - Option C.

Managed instance groups offer autoscaling capabilities that allow the automatic addition or deletion of instances from a managed instance group based on the increase or decrease in load. To create an autoscaler, it is required to specify the autoscaling policy and a target utilization level that the autoscaler will use to determine when to scale the group. Autoscaling can be done using the following policies - 

  • Average CPU utilization

  • HTTP load balancing serving capacity, which can be based on either utilization or requests per second.

  • Stackdriver monitoring metrics

In the given scenario, the autoscaling has been enabled by using Average CPU utilization autoscaling policy. So, the autoscaler will treat the target CPU utilization level as a fraction of the average use of all CPUs over time in the instance group. The autoscaler will add/start other instances if the average usage of the total CPUs exceeds the target utilization.

Here, the target CPU utilization is set to be 70% which means the autoscaler will have to maintain an average usage of 70% among all the CPUs in the instance group. But with one of the instances, it is found that CPU utilization is 75% while there are 4 instances available in the instance group right now. Considering all the instances, the average CPU usage will be less than 70%, and so the autoscaling won’t be enabled and no other instance will be added or deleted.

Option A is incorrect. The 60s is the default cooldown period that is alotted to allow the instances to end up initialization before the autoscaler begins to collect information from them. But as the information doesn’t affect the autoscaler decisions, it could not be the reason for not starting another instance in the given scenario.

Option B is incorrect. Autoscaling is active in the given scenario with the average CPU utilization autoscaling policy. So, ‘autoscaling is not active’ is not the correct option.

Option C is correct. As given, the CPU utilization is 75% when connected to one of the instances. As there are 4 instances available right now in the given managed instance group, the average CPU usage of the instance group will be less than 70%. As the autoscaling will be enabled only when the average CPU usages exceed the 70% limit, less average CPU usage is the reason for not starting another instance in the given scenario.

Option D is incorrect. The average CPU usage of the instance group is less than 70% - is the most possible reason for not starting another instance in the given scenario, so option C is the correct answer. Therefore, ‘none of the above’ can’t be the right answer and so, this is an incorrect choice.

Hence option C is correct whereas options A, B, and D are incorrect.

 

Reference Link:

https://cloud.google.com/compute/docs/autoscaler/scaling-cpu-load-balancing

Option B is incorrect. The command “gcloud compute networks subnets expand-ip-range subnetace1 --size=-12” is not correct. There is no --size parameter used in the expand-ip-range command; it should be prefix-length.

Option C is incorrect. The command “gcloud compute networks subnets expand-ip-range subnetace1 --prefix-length=-12” is not correct. Though the prefix length specified in the expand-ip-range command is less than 18 (as mentioned above), the prefix length cannot be negative.
